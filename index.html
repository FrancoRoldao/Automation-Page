Afinal, o que é teste e QA?

Embora as pessoas geralmente usem os termos "teste" e "garantia da qualidade" (QA) de forma intercambiável, teste e QA não são a mesma coisa. Teste é voltado para o produto, enquanto QA é voltado para o processo.

Teste

O teste de software é um conjunto de atividades para descobrir defeitos e avaliar a qualidade dos artefatos de software. Esses artefatos, quando estão sendo testados, são conhecidos como objetos de teste.
Vale lembrar que teste também inclui outras atividades e deve estar alinhado com o ciclo de vida de desenvolvimento de software.

QA

A QA é uma abordagem preventiva e orientada para o processo que se concentra na implementação e no aprimoramento dos processos. Ela funciona com base no fato de que, se um bom processo for seguido corretamente, ele gerará um bom produto. A QA se aplica aos processos de desenvolvimento e teste e é responsabilidade de todos em um projeto.

Os 7 princípios do Teste de Software.

Ao longo dos anos, foram sugeridos vários princípios de teste que oferecem diretrizes gerais aplicáveis a todos os testes. Descrição conforme syllabus:

O teste mostra a presença, não a ausência de defeitos

Os testes podem mostrar que os defeitos estão presentes no objeto de teste, mas não podem provar que não há defeitos (Buxton 1970). Os testes reduzem a probabilidade de defeitos não serem descobertos no objeto de teste, mas, mesmo que nenhum defeito seja encontrado, os testes não podem provar a corretude do objeto de teste.

Testes exaustivos são impossíveis

Testar tudo não é viável, exceto em casos triviais (Manna 1978). Em vez de tentar testar exaustivamente, as técnicas de teste, a priorização de casos de teste e os testes baseados em riscos devem ser usados para concentrar os esforços de teste.

Testes antecipados economizam tempo e dinheiro

Os defeitos que são removidos no início do processo não causarão defeitos subsequentes nos produtos de trabalho derivados. O custo da qualidade será reduzido, pois menos falhas ocorrerão posteriormente no SDLC (Boehm, 1981). Para encontrar defeitos logo no início, os testes devem ser iniciados o mais cedo possível.

Os defeitos se agrupam

Um pequeno número de componentes do sistema geralmente contém a maioria dos defeitos descobertos ou é responsável pela maioria das falhas operacionais (Enders 1975). Esse fenômeno é uma ilustração do Princípio de Pareto. Os agrupamentos de defeitos previstos e os agrupamentos de defeitos reais observados durante o teste ou em operação são uma entrada importante para o teste baseado em risco.

Os testes se degradam

Se os mesmos testes forem repetidos muitas vezes, eles se tornarão cada vez mais ineficazes na detecção de novos defeitos (Beizer 1990). Para superar esse efeito, talvez seja necessário modificar os testes e os dados de teste existentes, e talvez seja necessário escrever novos testes. Entretanto, em alguns casos, a repetição dos mesmos testes pode ter um resultado benéfico, por exemplo, em testes de regressão automatizados.

Os testes dependem do contexto

Não existe uma única abordagem universalmente aplicável aos testes. Os testes são feitos de forma diferente em contextos diferentes (Kaner 2011).

Falácia da ausência de defeitos

É uma falácia (ou seja, uma concepção errônea) esperar que a verificação do software garanta o sucesso de um sistema. Testar exaustivamente todos os requisitos especificados e corrigir todos os defeitos encontrados ainda pode produzir um sistema que não atenda às necessidades e expectativas dos usuários, que não ajude a atingir os objetivos de negócio do cliente e que seja inferior a outros sistemas concorrentes. Além da verificação, a validação também deve ser realizada (Boehm, 1981).

O Selenium é um conjunto de ferramentas de automação de código aberto para testar aplicativos web. Ele permite que os profissionais automatizem as interações com as páginas web, simulando ações de um usuário

Sabia que existem vários componentes no ecossistema do Selenium? Te convido a pesquisar sobre os componentes abaixo:

- Selenium WebDriver: Fornece a API que permite a automação de ações em navegadores.

- Selenium IDE: Uma extensão do navegador que permite gravar ações executando manualmente e executá-las posteriormente de forma automatizada.

- Selenium Grid: É uma ferramenta que permite a execução de testes paralelamente em vários navegadores e sistemas operacionais.

- Selenium Client Libraries: É a biblioteca que permite que os desenvolvedores escrevam os scripts em suas linguagens preferidas.

O JUnit é um framework de teste unitário amplamente utilizado na linguagem de programação Java. Sua principal função é facilitar a criação e execução de testes automatizados para verificar o funcionamento de componentes individuais do código Java, como classes e métodos.

Informações mais detalhadas sobre o JUnit:

- Teste Unitário: Testes unitários são responsáveis por verificar se partes específicas do código (geralmente funções ou métodos) produzem os resultados esperados quando dados de entrada são fornecidos.

- Estrutura de Testes: O JUnit fornece uma estrutura e um conjunto de anotações (como @Test) que permitem aos desenvolvedores definir testes de maneira organizada e clara. Os testes são escritos em métodos Java e podem ser agrupados em classes de teste.

Asserções: O JUnit oferece uma variedade de métodos de asserção (como assertEquals, assertTrue, assertNotNull, etc.) que fazem uma etapa do teste passar ou falhar, dependendo do resultado esperado/obtido.

Execução Automatizada: O JUnit é capaz de executar todos os testes automaticamente.

Relatórios de Teste: O JUnit gera relatórios detalhados após a execução dos testes, indicando quais testes passaram, quais falharam e por quê. Isso facilita a identificação e correção de problemas no código.

O Cucumber é um framework de automação de testes de software que segue a metodologia de desenvolvimento orientado por comportamento (BDD - Behavior-Driven Development). O principal objetivo do Cucumber é permitir que equipes de desenvolvimento e negócios colaborem na definição e automação de requisitos funcionais do software de forma mais legível e compreensível.

Principais características e componentes do Cucumber:

- Linguagem Gherkin: O Cucumber utiliza a linguagem Gherkin, que é uma linguagem de alto nível e legível por humanos para descrever o comportamento do software. Os cenários de teste são escritos em formato semelhante a uma linguagem natural, com palavras-chave como "Dado", "Quando" e "Então". Isso torna os cenários de teste compreensíveis

- Step Definitions: Os cenários de teste escritos em Gherkin são vinculados a código real por meio de definições de passos (step definitions).

- Reutilização de Passos: O Cucumber permite a reutilização de etapas comuns em vários cenários, o que torna a escrita de testes mais eficiente e ajuda na manutenção.

- Integração com Ferramentas de Teste e CI/CD: O Cucumber pode ser facilmente integrado a outras ferramentas de automação de teste, bem como a sistemas de integração contínua e entrega contínua (CI/CD).

O Appium é um framework de automação de teste que é usado principalmente para testar aplicativos móveis em diferentes plataformas, como iOS, Android e Windows.
Ele permite que os desenvolvedores e testadores escrevam scripts de teste automatizados que interagem com os aplicativos móveis.

Ai vão alguns detalhes sobre o Appium:

- Linguagens de Programação: Você pode escrever scripts de automação no Appium em várias linguagens de programação populares.

- Abordagem Baseada em WebDriver: O Appium utiliza o protocolo WebDriver para automatizar a interação com os aplicativos. Isso significa que ele segue o padrão de automação da web, tornando-o mais familiar para muitos.

- Acesso a Recursos do Dispositivo: O Appium oferece acesso a recursos do dispositivo móvel, como câmera, GPS, acelerômetro e sensores, permitindo testes mais abrangentes e realistas.

Suporte a Emuladores e Dispositivos Físicos: Você pode usar o Appium tanto com emuladores quanto com dispositivos físicos.

- Integração com Estruturas de Teste: O Appium pode ser integrado a várias estruturas de teste e ferramentas de automação, como o Selenium, o TestNG e o JUnit.

A biblioteca Rest Assured é uma biblioteca Java amplamente utilizada para automatizar testes de APIs REST (Representational State Transfer). Ela simplifica a tarefa de criar e executar testes automatizados em serviços web que seguem os princípios de design RESTful, como endpoints de API que respondem a solicitações HTTP.

Aqui está uma visão geral do que a biblioteca Rest Assured é e o que ela faz:

- Automação de Testes de API: O Rest Assured é projetado especificamente para automatizar testes de APIs RESTful. Ele fornece uma maneira eficiente e expressiva de interagir com APIs por meio de solicitações HTTP (GET, POST, PUT, DELETE, etc.) e verificar as respostas obtidas.

- Validação de Respostas: Rest Assured oferece métodos convenientes para comparação e verificação dos valores retornados pelas APIs.

- Integração com Estruturas de Teste: O Rest Assured pode ser facilmente integrado a estruturas de teste populares, como JUnit e TestNG, para organizar e executar testes de API de maneira eficaz.

- Testes em Cadeia (Chaining): O Rest Assured permite a criação de sequências encadeadas de comandos, tornando possível encadear várias verificações e interações em uma única solicitação HTTP. Isso simplifica a criação de testes complexos.

- Geração de Relatórios: Você pode gerar relatórios de testes detalhados usando o Rest Assured.

Tipos de inputs HTML

A tag input em HTML é um elemento de formulário usado para criar campos de entrada interativos em uma página da web. Esses campos de entrada podem ser usados para coletar informações do usuário, como texto, números, datas, seleções, etc. A tag input é um elemento vazio, o que significa que não tem uma tag de fechamento. Em vez disso, as informações sobre o tipo de entrada e o valor são especificadas usando atributos.
